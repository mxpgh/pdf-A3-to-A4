# 折腾一个A3转A4工具，我经历了什么

上周接了个活：一个能把A3试卷自动拆成A4的桌面工具。

听起来挺简单对吧？代码写好了，界面画好了，连安装包都打好了。按理说已经是"做完"的状态了。

结果用户说："继续。"

那就继续呗。然后就是一轮又一轮的"改-打包-试-再改"循环。前后折腾了五个回合，记个流水账。

---

**第一回合：启动就崩**

拿到代码先跑一下。点了预览按钮——

`fitInView() takes 1 positional argument but 2 were given`

翻译成人话：我写了个函数叫 `fitInView`，但 Qt 的父类里也有个同名函数。我想调父类的，结果调成自己的，参数对不上，炸了。

改名搞定。三行代码的事。

**第二回合：按钮把内容盖了**

用户发来一张截图。预览窗口确实出来了，但底部的按钮直接把预览画面挡住了一半。

这是我布局没写好。Qt 的布局系统里，你得明确告诉它"哪个控件该撑大，哪个控件该固定"。预览窗口没设权重，结果按钮区把它挤扁了。

改了几个参数：预览区 `stretch=1`，去掉多余的弹性空间，压缩一下标题和按钮的高度。

搞定。重新打包。

**第三回合：转出来的PDF不好看**

"转换没问题，预览也没问题，但转出来的格式看起来有点乱。"

我检查了输出：尺寸对、页数对、分割线位置也对。但所有内容都**顶到纸边**，没有任何边距。

这谁打印出来谁崩溃。

我加了个边距参数，默认30pt（大概1厘米）。但又觉得这方案太死板——万一别的文档边距不一样呢？

用户比我清醒："能不能自动检测左右两边的空白？"

行，写个自动裁剪函数：把图片转灰度，逐列扫描，找到真正有内容的区域，剪掉多余白边，保留一点内边距，然后居中放到A4纸上。不管原文档留白多少，输出都一样整洁。

**第四回合：分割线能不能再聪明点**

"预览分割线能不能自动定位到两页中间的那个空白区？"

老实说，原来的算法找的是"最暗的竖线"——对有装订折痕的扫描件有效。但现在用户拿来测的是印刷清晰的A3试卷，中间是白白净净的留白，根本没有暗线。

得让算法同时处理两种情况：
- 有折痕 → 找最暗的位置
- 是留白 → 找最亮的位置

怎么判断？看中间区域的内容密度——中间比两边暗，就是折痕；中间比两边亮，就是留白。自动切换策略，再加个高斯权重让分割点别跑偏。

改完一试，两种文件都能精确定位。

**第五回合：一个487MB的安装包**

"包太大了，优化下。"

我一看：487MB。一个分割PDF的工具，快半G了。

查一下安装包里面有什么——好家伙，scipy 112MB、pyarrow 92MB、pandas 20MB……全是 numpy 的"附带亲戚"，我一个都没用到。

问题出在打包脚本写了个 `--collect-all numpy`，意思是"把 numpy 全家老小都带上"。

改掉：去掉无脑全包，改成只导入确实需要的模块，再手动排除那些庞然大物。加上 `--strip` 去去 debug 符号。

487MB → 217MB。省了一半多，功能一切正常。

---

**后记**

有些感慨。

如果一开始我就问"你想要什么"，用户可能也说不出太具体的东西。很多需求是**看到实物之后才浮现的**——"哦，原来它是这样的，那这里能不能改一下？"

一圈圈改下来，从能用到好用，需要的不是多高明的技术，就是愿意一轮轮折腾的耐心。

另外就是：不要轻易写 `--collect-all`。
